**Wave ordering:** depends on Plan 02-02 because the workflow's new step executes the `export_to_json.py` file landed in 02-02. Does NOT depend on Plan 02-01 — json_repo lands separately on the client side and is not exercised until Plan 02-04.

**Post-Wave-2 state:** CDN is live, published, and reachable. No client consumes it yet. The supabase import chain is still on the runtime path of `backend.py` — that stays until Plan 02-04.

# THREAT MODEL

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions secrets → export step | SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY must only be in env: of the export step; never echoed. |
| peaceiris action → gh-pages branch | Uses `${{ secrets.GITHUB_TOKEN }}` with `contents: write` scope. No third-party creds. |
| Public GitHub Pages → world | No access control; anyone can download the JSON. No PII or secrets in the data. |
| Operator browser → GitHub repo Settings | UI action; auditable via repo's audit log. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-21 | I (Info disclosure) | Supabase service-role key in workflow logs | mitigate | Pass key via env: from secrets; GitHub Actions auto-redacts known secret values in logs. Reviewed by `grep -iE 'SUPABASE.*:' update-dataset.yml` — must be `${{ secrets.* }}` not a literal. |
| T-02-22 | T (Tampering) | Malicious PR modifies workflow to exfiltrate secrets | mitigate | GitHub's default: secrets not available to workflows triggered by pull_request from forks. The workflow triggers on schedule + workflow_dispatch only (existing design), which limits exposure. |
| T-02-23 | E (Elevation of privilege) | `contents: write` permission scope | accept | Minimum required for peaceiris push to gh-pages. Narrower than the default `contents: write + issues: write + ...` full scope. |
